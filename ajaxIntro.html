Ajax in a nutshell.

Creating a modern web application requires an interrupted flow much like the one offered by the mayority of Desktop based application where you expect the display contents of your programs to remain visible while the interfce elements continue to responding to the users commands, as well as all computing processes happening in the background without being noticed.

Why is Ajax so useful? The Ajax framework creates this extra functionality for processing between the web page and the server. It intercepts any request from the user and handles the response from the server asynchronously.

jQuery to the rescue.

Implementing Ajax from scratch is tedious work. Fortunately for us jQuery solves the issue with jQuery methods for performing Ajax calls to the server

At this point if you have any doubts about Callbacks, I recommend you spend some time refreshing this concept as it is the base of Ajax and Node.js

Remember, all of this is javaScript! Even if we'll end up using the jQuery library we must understand the basic undelying concept behind it. Microsoft developed the XMLHttpRequest object to interact with servers. It was later adopted by Apple, Modzila and Google. The XMLHttpRequest constructor must be called before any other method calls like this: var myRequest = new XMLHttpRequest();

The jQuery XMLHttpRequest (jqXHR) object is a superset of the browser's native XMLHttpRequest object. jQuery's $.ajax() function underlies all Ajax requests sent to the server. The jqXHR objects returned by $.ajax() implements the Promise interface, giving them all the properties, methods, and behavior of a Promise (see Deferred object for more information). These methods take one or more function arguments that are called when the $.ajax() request terminates. This allows you to assign multiple callbacks on a single request, and even to assign callbacks after the request may have completed. More importantly, we can use other jQuery DeferredObject methods like .done(), .fail(), .always(), .then()
